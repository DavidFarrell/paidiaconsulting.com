Phase one in more detail

01 · Nothing irreversible

The pilot uses conventional, replaceable parts. The real data-integration architecture stays at months 6-12 - the pilot de-risks it first, rather than committing to it early.

  • Quick and dirty data grabs
  • Wizard of Oz where appropriate
  • Noticing where performance bottlenecks lie

02 · Instrument from day one

Every LLM call is logged (using tracing systems):

  • LangSmith
  • Arize Phoenix
  • Braintrust

The goal is to give us a dataset to optimise prompts and model choices as we move from experimentation to implementation.

  • This will allow us to move from fat models to cheap / open models

03 · The interface

Working assumption: a Cowork interface + skills + MCPs thrown together quickly for the initial stages, but firmed up from the midway point so it can survive rigorous architecture in the next phase.

Push the briefing

The plan has the briefing arrive when an account manager asks for it. We could also push it - have it land ahead of a scheduled call, triggered from the calendar, before anyone thinks to pull it.

• Same record, same loop - just a second trigger. Worth doing once the on-demand briefing has proven itself, not before.
Appendix A2

Raw transcripts aren't mostly noise

The instinct is to extract the signal and discard the transcript. I'd keep the raw transcript - redacted - rather than throw it away up front. The off-hand line that turns out to matter is usually in the raw text, not the summary.

• We don't have to argue it - prove it with an A/B: a briefing built with the raw-transcript signal against one built without, and see which an account manager prefers.
